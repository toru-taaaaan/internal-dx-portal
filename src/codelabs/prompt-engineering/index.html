このハンズオンでは、AIへの指示文（プロンプト）を上手に書くための技術「プロンプトエンジニアリング」を学びます。

難しい知識は不要です。いくつかのパターンを知るだけで、AIからの回答品質が大幅に向上します。

このコースで学ぶこと

  • プロンプトを構成する4つの要素
  • ゼロショット vs フューショット（例示）の違い
  • 思考の連鎖（Chain-of-Thought）で精度を上げる方法
  • ロールプロンプティングでトーンをコントロールする方法
  • よくある失敗パターンと改善法
  • 反復改善のワークフロー

対象者

  • GeminiやChatGPTを使い始めたが、思い通りの回答が得られないと感じている方
  • AI Champion として社内でプロンプトのコツを伝えたい方
  • プログラミング経験は不要

所要時間

約 45分

使用するツール

  • Gemini（gemini.google.com）またはお好みのAIチャット
  • ブラウザのみ。インストール不要

Googleのプロンプト設計ガイドでは、すべてのプロンプトは次の4つの要素で構成されると定義しています。

要素 説明 必須？
タスク AIに何をしてほしいか。指示の核心部分。 必須
システム指示 AIの役割・トーン・制約を最初に設定する。 任意
例示（フューショット） 期待する入出力のサンプルを見せる。 任意
コンテキスト 背景情報・参照データ・前提条件を与える。 任意

4要素を使った例

会議の議事録を要約させる場合：

【システム指示】 あなたは情報システム部のアシスタントです。 簡潔で正確な日本語で回答してください。 【タスク】 以下の議事録を3つの箇条書きに要約してください。 決定事項・次のアクション・期限を必ず含めること。 【コンテキスト】 （ここに議事録のテキストを貼り付ける）
ポイント：4要素を全部使う必要はありません。まずタスクだけで試し、回答が期待と違ったら他の要素を加えていきましょう。

試してみよう

Gemini を開き、タスクだけの指示と4要素を揃えた指示を比べてみましょう。

タスクのみ：

先週の会議を要約して

4要素を使った版：

あなたは50代の部長向けに資料を作る社内アシスタントです。 以下の会議メモを3点の箇条書きに要約してください。 ・決定事項 ・次のアクションと担当者 ・期限 【会議メモ】 （メモをここに貼る）

回答の具体性が大きく変わることを確認してください。

AIへの指示に「例」を含めるかどうかで回答品質が変わります。

ゼロショット（例なし）

例を一切示さず、タスクだけを伝える方法です。シンプルですが、フォーマットが不安定になりがちです。

次のメールを丁寧な日本語に書き直してください。 「明日の会議ですが、参加できなくなりました。すみません。」

フューショット（例あり）

「このような入力→この形式で出力」という例を2〜4件示す方法です。Googleのガイドでは常にフューショット例を含めることを推奨しています。

メールを丁寧な敬語に書き直します。 【例1】 入力: 明日の会議、遅れます。 出力: 明日の会議ですが、やむを得ない事情により到着が遅れる見込みです。大変申し訳ございません。 【例2】 入力: 資料送ってください。 出力: お手数をおかけしますが、ご都合がよろしければ関連資料をご送付いただけますでしょうか。 【タスク】 入力: 明日の会議ですが、参加できなくなりました。すみません。 出力:
フューショットのコツ：
  • 例は2〜4件が目安（多すぎると逆効果）
  • 「やってはいけない例」より「正しい例」を見せる
  • 例のフォーマットを統一する
  • 実際の業務で使うパターンを例にすると効果大

試してみよう

次の2つを同じAIに送り、回答の違いを比べてください。

ゼロショット版：

このメールの件名を考えてください。 「IIJとのSD-WAN契約について、来月末までに社内承認が必要です。今週中に担当部署へ確認をお願いします。」

フューショット版：

メール本文から件名を生成します。件名は20文字以内で、要件番号・内容・期限の形式で。 例1: 本文「NW機器の保守期限が来月末に迫っています」→ 件名「【要対応】NW機器保守期限確認（来月末）」 例2: 本文「新入社員のアカウント作成依頼が届いています」→ 件名「【依頼】新入社員アカウント作成（今週中）」 本文: 「IIJとのSD-WAN契約について、来月末までに社内承認が必要です。今週中に担当部署へ確認をお願いします。」 件名:

複雑な問いや判断が必要なタスクでは、AIに「ステップを追って考えさせる」ことで精度が上がります。これを Chain-of-Thought（CoT） と呼びます。

CoTが効くケース

  • 費用計算・数値比較
  • 複数の選択肢の優劣判断
  • 論理的な意思決定フロー
  • リスクの洗い出し

CoTなし（直接回答を求める）

SD-WAN 3社のうちどれが最もコスト効率が良いですか？ A社: 初期費用100万円、月額15万円 B社: 初期費用0円、月額25万円 C社: 初期費用50万円、月額18万円

CoTあり（ステップを追って考えさせる）

SD-WAN 3社のコスト効率を比較します。ステップを追って計算してください。 A社: 初期費用100万円、月額15万円 B社: 初期費用0円、月額25万円 C社: 初期費用50万円、月額18万円 【手順】 1. 各社の3年間（36ヶ月）のトータルコストを計算する 2. 初期費用を月割りした実質月額を求める 3. 3社を安い順にランキングする 4. 最終的な推薦とその理由を1文で述べる
CoTを引き出すフレーズ：
  • 「ステップを追って考えてください」
  • 「手順を番号で示してください」
  • 「判断の根拠を明示してください」
  • 「まず〇〇を確認し、次に…という順序で」

試してみよう

業務上の判断が必要な質問を1つ考え、CoTなし・ありの両方で送ってみてください。回答の根拠の明確さがどう変わるか確認しましょう。

AIに「あなたは〇〇です」と役割を与えることで、回答のトーン・専門性・語彙が変わります。

ロールなし

ゼロトラストとは何ですか？

ロールあり（技術者向け）

あなたはネットワークセキュリティの専門家です。 CISOへの技術説明を想定して、ゼロトラストの概念とSASEとの違いを説明してください。

ロールあり（非エンジニア向け）

あなたは社内IT研修の講師です。 プログラミング経験のない50代の部長に向けて、ゼロトラストとは何かを身近な例え（会社の入館管理など）を使って説明してください。200文字以内で。

ビジネスで使えるロール例

役割 使いどころ
「プロの編集者」 文章の校正・読みやすさ改善
「弁護士」「法務担当」 契約書・規約のリスク確認（※最終確認は専門家に）
「50代の部長」 資料を受け取る側の視点でフィードバックを得る
「厳格なレビュアー」 提案書の弱点・反論を洗い出す
「IT未経験の社員」 マニュアルの分かりやすさを検証する
注意：ロールを与えると回答が変わりますが、事実確認は必ず別途行ってください。AIは役割に合わせて自信を持って回答しますが、誤情報が含まれることがあります。

試してみよう

同じ質問を「専門家向け」と「非専門家向け」の2パターンのロールで送り、回答の違いを比べてください。

Googleのプロンプト設計ガイドに記載されているよくある失敗パターンと改善法を紹介します。

失敗1：曖昧な要求

❌ 曖昧
「この文章をもっと良くしてください」
✅ 具体的
「この文章を、50代の部長が3分で読めるように、体言止めで箇条書き3点にまとめてください」

失敗2：複数のタスクを一度に依頼

❌ 詰め込みすぎ
「この資料を要約して、問題点を指摘して、改善案を書いて、メールの件名も考えてください」
✅ 分割して依頼
まず要約 → 次に問題点 → 次に改善案 → 最後に件名、と順番に送る

失敗3：感情的なプレッシャー

❌ 効果なし
「これはとても重要です！絶対に正確に答えてください！！」
✅ 有効な制約
「回答は事実のみを含め、不確かな情報には『要確認』と明記してください」

失敗4：否定形の指示

❌ 否定形
「専門用語を使わないでください」
✅ 肯定形
「中学生でも理解できる言葉で説明してください」

失敗5：矛盾した指示

❌ 矛盾
「詳しく説明してください。ただし100文字以内で。」
✅ 一貫性
「要点を3点に絞り、各点を1文（50文字以内）で説明してください」
チェックリスト：プロンプトを送る前に確認しましょう。
  1. 期待する出力フォーマット（箇条書き・表・文章）を明記したか？
  2. 出力の長さや制約（文字数・件数）を書いたか？
  3. 対象読者・トーンを指定したか？
  4. タスクは1つに絞ったか？

プロンプトは1回で完成させようとしないことが大切です。Googleは以下の反復ワークフローを推奨しています。

プロンプト改善の6ステップ

  1. 目標を定める — 「どんな回答が来たら成功か」を先に決める
  2. 最初のプロンプトを書く — タスクだけの簡単なものでOK
  3. 回答を評価する — 目標と比べてどこがズレているか？
  4. 失敗モードを特定する — 内容の間違い / フォーマットの問題 / トーンの違い / 不要な情報 / 回答拒否
  5. 対策を施して再試行 — 失敗モードに合った要素を追加する
  6. 繰り返す — 目標を満たすまでステップ3〜5を繰り返す

失敗モード別の対策

失敗モード 対策
内容が間違っている コンテキスト（参照情報）を追加する
フォーマットが期待と違う 出力形式を明示する / フューショット例を追加する
トーンが合わない ロールと読者を指定する
長すぎる / 短すぎる 文字数・件数の制約を明記する
的外れな回答 タスクを分割する / CoTで手順を指定する
現場のコツ：うまくいったプロンプトは「プロンプトテンプレート集」としてGoogleドキュメントやスプレッドシートに保存しておきましょう。チームで共有することで組織全体の生産性が上がります。
