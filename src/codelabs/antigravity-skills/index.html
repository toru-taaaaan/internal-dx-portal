このCodelabでは Google Antigravity の エージェントスキル を構築する方法を学びます。

スキルとは、専門知識とワークフローを使って AI エージェントの機能を拡張する 軽量のオープンフォーマット です。

このCodelabで学ぶこと

  • スキルがなぜ重要か（コンテキスト飽和の問題）を理解する
  • スキルのディレクトリ構造と SKILL.md の書き方を習得する
  • 5段階のスキル実装パターン（基本〜複合）を体験する
  • スクリプト・リファレンス・例を組み合わせた高度なスキルを作る

このCodelabで作るもの

5つのスキルを段階的に構築します：

  1. git-commit-formatter — コミットメッセージを Conventional Commits 形式に自動整形
  2. license-header-adder — 新規ファイルにライセンスヘッダーを自動付与
  3. json-to-pydantic — JSON を Pydantic モデルに変換
  4. database-schema-validator — SQL スキーマをルールに従って検証
  5. adk-tool-scaffold — ADK ツールクラスのひな形を自動生成

前提条件

  • Antigravity インストール済み（未インストールの場合は「Antigravity 使い方入門」を先に完了してください）
  • Google Workspace アカウント（会社の Gmail アドレス）
  • 所要時間：約 45 分

プログラミングの経験が浅くても大丈夫です。スキルの仕組みを理解することで、エージェントをより効果的に活用できるようになります。

問題：コンテキストの飽和

現代の AI エージェントは複雑な推論エンジンに進化しています。しかし、コードベース全体と数百のツールを 無差別に読み込む と、コンテキストの飽和 が発生します。

未使用のツール情報だけで 4〜5 万トークン を消費し、次の課題が生じます：

  • レイテンシの増加（応答が遅くなる）
  • コストの無駄
  • コンテキスト劣化（関連性のないデータで推論が混乱する）

解決策：プログレッシブ開示

スキルは「プログレッシブ開示」アーキテクチャを実現します。

タイミング

読み込むもの

セッション開始時

スキル名と説明だけの軽量メニュー

ユーザーの意図が一致したとき

重い手続き型知識（SKILL.md 本文）を動的に読み込む

これにより、数十のツールを使う大規模プロジェクトでも パフォーマンスが維持 されます。

スキルの主な特徴

オンデマンド機能拡張

システムプロンプト（常に読み込まれる）と異なり、スキルはエージェントが ユーザーのリクエストに関連する と判断した場合にのみ読み込まれます。

実行能力

スキルは Python や Bash スクリプトをバンドルでき、エージェントにローカルマシンや外部ネットワークで 複雑な複数ステップアクションを実行 させることができます。

ルール・ワークフローとの違い

種類

特徴

ルール

受動的・常に有効なガードレール

ワークフロー

能動的・ユーザーがトリガーするマクロ

スキル

エージェントトリガー：意図を自動検出し専門知識を動的提供

スキルと MCP の関係：MCP はエージェントの「手」（外部システムへの接続）、スキルはそれを指示する「脳」（方法論のパッケージ）です。

スキルの構造

Antigravity でのスキルは、定義ファイル（SKILL.md） とオプションのサポート資産を含む ディレクトリベースのパッケージ です。

2つのスコープ

  1. ワークスペーススコープ：<workspace-root>/.agent/skills/
    プロジェクト固有のスクリプト（デプロイ、データベース管理など）に適切
  2. グローバルスコープ：~/.gemini/antigravity/skills/
    すべてのプロジェクトで使用可能。汎用ユーティリティに適切

一般的なディレクトリ構造

my-skill/
    ├── SKILL.md              # 定義ファイル（必須）
    ├── scripts/              # [オプション] Python・Bash・Node スクリプト
    │   ├── run.py
    │   └── util.sh
    ├── references/           # [オプション] ドキュメント・テンプレート
    │   └── api-docs.md
    └── assets/               # [オプション] 静的資産

SKILL.md 定義ファイル

スキルの「脳」となるファイルで、2部構成 です。

① YAML フロントマター（メタデータ）

エージェントのルーターによってインデックスされる軽量なメタデータです。

---
name: database-inspector
description: Use this skill when the user asks to query the database, check table
schemas, or inspect user data in the local PostgreSQL instance.
---

主要フィールド：

  • name：スコープ内で一意（小文字・ハイフン使用可。例：postgres-query）
  • description：最も重要な必須フィールド。LLM がセマンティック関連性を認識できるほど説明的であるべき。「読み取り専用 SQL 実行」のような曖昧な記述では不十分です

② マークダウン本文（手続き型指示）

スキルが有効になるとエージェントのコンテキストウィンドウに挿入されます。含めるべき内容：

  1. 目標：スキルで達成できる内容
  2. 手順：ステップごとのロジック
  3. 例：少数ショットの入出力例
  4. 制約：「してはいけない」ルール

スクリプト統合

スキルの最強機能の1つは スクリプト実行への委任 です。LLM が直接行えない操作（バイナリ実行・複雑な計算・レガシーシステム連携）を可能にします。スクリプトは scripts/ サブディレクトリに配置し、相対パスで参照します。

description フィールドは最も重要です。「データベースツール」のような曖昧な説明ではなく、「ユーザーがデータベースへのクエリ、テーブルスキーマの確認、ユーザーデータの検査を求めるときに使用する」のように具体的に書きましょう。

レベル 1：基本ルーター（git-commit-formatter）

「Hello World」スキルです。開発者のあいまいなコミットメッセージ（「wip」「fix bug」など）を Conventional Commits 仕様 に従わせます。

まず、ワークスペースの .agent/skills/git-commit-formatter/ ディレクトリを作成し、SKILL.md を以下の内容で作成します：

---
name: git-commit-formatter
description: Formats git commit messages according to Conventional Commits
specification. Use this when the user asks to commit changes or write a commit message.
---

# Git Commit Formatter Skill

When writing a git commit message, you MUST follow the Conventional Commits
specification.

## Format
`<type>[optional scope]: <description>`

## Allowed Types
- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that do not affect the meaning of the code
- **refactor**: A code change that neither fixes a bug nor adds a feature
- **perf**: A code change that improves performance
- **test**: Adding missing tests or correcting existing tests
- **chore**: Changes to the build process or auxiliary tools

## Instructions
1. Analyze the changes to determine the primary `type`.
2. Identify the `scope` if applicable.
3. Write a concise `description` in an imperative mood.
4. If there are breaking changes, add a footer starting with `BREAKING CHANGE:`.

## Example
`feat(auth): implement login with google`

動作確認：チャットで「Commit these changes」と入力すると、エージェントが Conventional Commits 形式のコミットメッセージを提案します。

レベル 2：アセット利用（license-header-adder）

「参照」パターンです。重い静的テキストをプロンプトに直接入れるのは無駄です。resources/ フォルダにオフロードします。

license-header-adder/
├── SKILL.md
└── resources/
    └── HEADER_TEMPLATE.txt

SKILL.md は必要な場合にのみテンプレートファイルを読み取るようエージェントに指示します：

---
name: license-header-adder
description: Adds the standard open-source license header to new source files.
Use involves creating new code files that require copyright attribution.
---

# License Header Adder Skill

## Instructions
1. **Read the Template**:
   First, read the content at `resources/HEADER_TEMPLATE.txt`.

2. **Prepend to File**:
   When creating a new file (.py, .java, .js, .ts, .go),
   prepend the target_file content with the template content.

3. **Modify Comment Syntax**:
   - C-style languages (Java, JS, TS, C++): Keep `/* ... */` block as is.
   - Python, Shell, YAML: Convert to use `#` comments.
   - HTML/XML: Use `<!-- ... -->`.

静的テキストをプロンプトに直接入れると毎回トークンが消費され、モデルが法的テキストをハルシネーションする可能性があります。references/ フォルダへのオフロードが最善策です。

レベル 3：例による学習（json-to-pydantic）

「少数ショット」パターンです。LLM はパターンマッチングエンジンなので、詳細な指示より 優れた例（Input → Output） を示す方が効果的です。

json-to-pydantic/
├── SKILL.md
└── examples/
    ├── input_data.json      # Before State
    └── output_model.py      # After State

input_data.json：

{
    "user_id": 12345,
    "username": "jdoe_88",
    "is_active": true,
    "preferences": {
        "theme": "dark",
        "notifications": ["email", "push"]
    },
    "last_login": "2024-03-15T10:30:00Z",
    "meta_tags": null
}

output_model.py：

from pydantic import BaseModel, Field
from typing import List, Optional

class Preferences(BaseModel):
    theme: str
    notifications: List[str]

class User(BaseModel):
    user_id: int
    username: str
    is_active: bool
    preferences: Preferences
    last_login: Optional[str] = None
    meta_tags: Optional[List[str]] = None

動作確認：JSON スニペットを貼り付けて「Convert this JSON to a Pydantic model」と入力すると、エージェントが examples/ ペアを確認して同じスタイルで Pydantic クラスを生成します。

レベル 4：手続き型ロジック（database-schema-validator）

「ツール使用」パターンです。LLM の推測ではなく、決定論的なスクリプト に検証を委任します。

database-schema-validator/
├── SKILL.md
└── scripts/
    └── validate_schema.py

スキルの SKILL.md はスクリプト実行を指示します：

python scripts/validate_schema.py <path_to_user_file>

スクリプトが exit code 0 を返せば「検証OK」、exit code 1 を返せばエラーメッセージを報告します。検証ポリシー：

  1. 安全性：DROP TABLE 禁止
  2. 命名規則：テーブル名は snake_case
  3. 構造：すべてのテーブルに id 主キーが必要

レベル 5：アーキテクト（adk-tool-scaffold）

これまでのすべての機能を統合した最も複雑なパターンです。

adk-tool-scaffold/
├── SKILL.md
├── resources/
│   └── ToolTemplate.py.hbs    # Jinja2 テンプレート
├── scripts/
│   └── scaffold_tool.py       # ジェネレータースクリプト
└── examples/
    └── WeatherTool.py         # 参考実装

ADK（Agent Development Kit）用の新しいツールクラスを 自動生成 します。スキルが実行する手順：

  1. ツール名を抽出（例：「StockPrice」「EmailSender」）
  2. examples/WeatherTool.py で ADK ツールの構造を確認
  3. python scripts/scaffold_tool.py <ToolName> を実行してひな形ファイルを生成
  4. 生成されたファイルを編集して実装を完成させる

動作確認：「Create a new ADK tool called StockPrice to fetch data from an API」と入力すると、エージェントがスクリプトを実行し StockPriceTool.py を作成して examples/ を参照しながらガイダンスを提供します。

レベル5はスクリプト・テンプレート・例の3つを組み合わせています。複雑なタスクほどこの組み合わせが効果的です。

重要なポイント

  • description が最重要：LLM がセマンティックマッチングに使う。曖昧な説明はスキルが発動しない
  • 重いテキストはオフロード：静的テキストは references/ に、スクリプトは scripts/ に

  • スコープを使い分ける：プロジェクト固有はワークスペース、汎用はグローバル

スキルとMCPの関係

MCP はエージェントの「手」（GitHub や PostgreSQL などの外部システムへの接続）、スキルはそれを指示する「脳」（ツールの使い方をパッケージ化した軽量な知識）です。両者を組み合わせることで、組織のベストプラクティスと安全基準を厳守するスペシャリストエージェントが実現します。

  • スキルと仕様駆動開発を組み合わせてより高精度な開発を実践する
