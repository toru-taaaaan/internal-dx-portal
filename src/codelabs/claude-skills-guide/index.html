このCodelabはAnthropicが2026年1月に公開した公式マニュアル 「The Complete Guide to Building Skills for Claude」（全33ページ）を、完全に日本語で再現したものです。

スキルとは何か

スキル（Skill）とは、Claudeに特定のタスクやワークフローを教える指示のセットです。シンプルなフォルダとしてパッケージされています。スキルはClaudeをあなたのニーズに合わせてカスタマイズする最も強力な方法の一つです。

毎回の会話で好みやプロセス、ドメイン知識を説明し直す代わりに、一度教えれば毎回の会話で活用できます。

このガイドで学ぶこと

  • スキル構造の技術要件とベストプラクティス
  • スタンドアロンスキルとMCP強化ワークフローのパターン
  • 様々なユースケースで効果的と確認されたパターン
  • スキルのテスト・改善・配布方法

このガイドの対象者

  • Claudeに特定ワークフローを一貫して実行させたい開発者
  • Claudeに特定ワークフローを実行させたいパワーユーザー
  • 組織全体でClaudeの動作を標準化したいチーム

2つの読み方

スタンドアロンスキルを作る場合

「Fundamentals」「Planning and Design」「Testing and Iteration」を中心に読む。カテゴリ1〜2が該当。

MCP統合を強化する場合

「Skills + MCP」セクションとカテゴリ3が中心。両パスで技術要件は共通。

このガイドを読み終えると、skill-creatorを使って15〜30分で最初のスキルを作り上げられます。

スキルはAnthropicが設計したオープンなファイルフォーマットです。仕組みを理解することで、効果的なスキルを作れるようになります。

スキルフォルダの構成

スキルは以下の構造を持つフォルダです：

your-skill-name/
├── SKILL.md              # 必須 — メインのスキルファイル
├── scripts/              # オプション — 実行可能なコード
│   ├── process_data.py   # 例
│   └── validate.sh       # 例
├── references/           # オプション — ドキュメント
│   ├── api-guide.md      # 例
│   └── examples/         # 例
└── assets/               # オプション — テンプレートなど
    └── report-template.md # 例

SKILL.md さえあれば最小構成として動作します。name と description の2フィールドのみが必須です。

コア設計原則1：段階的開示（Progressive Disclosure）

スキルは3段階の読み込みシステムを採用しています。これが100個のスキルがあってもコンテキストウィンドウを圧迫しない理由です。

レベル

読み込む内容

説明

Level 1
YAML frontmatter

nameとdescriptionのみ

常にClaudeのシステムプロンプトに読み込まれる。どのスキルをいつ使うべきかを判断するのに必要な最小限の情報を提供する。

Level 2
SKILL.md本文

SKILL.mdの本文全体

Claudeが現在のタスクに関連すると判断したときに読み込まれる。完全な指示とガイダンスを含む。

Level 3
リンクされたファイル

スクリプト・参照ドキュメント等

スキルフォルダ内にバンドルされた追加ファイル。Claudeが必要と判断した場合のみ読み込まれる。

この段階的開示により、トークン使用量を最小化しながら専門的な知識を保持できます。

コア設計原則2：ポータビリティ（Portability）

スキルはClaude.ai、Claude Code、APIで同一に動作します。環境が必要な依存関係をサポートしていれば、修正なしにすべてのプラットフォームで動作します。

コア設計原則3：コンポーザビリティ（Composability）

Claudeは複数のスキルを同時に読み込めます。あなたのスキルは他のスキルと一緒にうまく動作するよう設計してください。自分が唯一の機能であると仮定しないでください。

MCP統合を行う場合：Skills + Connectors

すでに動作するMCPサーバーがあるなら、難しい部分は終わっています。スキルは知識レイヤーとして機能します。すでに知っているワークフローとベストプラクティスを取り込み、Claudeが一貫して適用できるようにします。

キッチンの比喩

MCP（Connectivity）

プロのキッチンを提供：ツール、食材、機器へのアクセス。Claudeをあなたのサービスに接続し、リアルタイムデータアクセスとツール呼び出しを可能にする。

Claudeが「できること」

+

Skills（Knowledge）

レシピを提供：価値あるものを作るためのステップバイステップの指示。Claudeにあなたのサービスを効果的に使う方法を教え、ワークフローとベストプラクティスを取り込む。

Claudeが「どうやるか」

スキルなしの場合 vs. スキルありの場合

スキルなし

• MCPに接続したが何をすればいいか分からない
• 「Xをどうやるか」というサポート問い合わせが増える
• 毎回の会話がゼロからスタート
• ユーザーごとに異なるプロンプトで一貫性がない
• ワークフロー指導の不足なのに、コネクターが非難される

スキルあり

• 必要なときにプリビルトワークフローが自動起動
• 一貫性のある信頼できるツール使用
• すべてのインタラクションにベストプラクティスが組み込み
• 統合の学習曲線が低下

MCPはあなたのサービスにアクセスするための「配線」、スキルはその配線を活用するための「知恵」です。両者は補完し合う関係です。

コードを書く前に、スキルが実現すべき具体的なユースケースを2〜3個特定してください。これがスキル設計の出発点です。

良いユースケースの定義

1
ユーザーが何を達成したいかを定義する 具体的なアウトカム（成果）から逆算する
2
必要なマルチステップワークフローを特定する 「このインプットが来たとき、どんな手順で処理するか」を整理する
3
必要なツールを確認する Claudeのビルトイン機能で済むか、MCPが必要かを判断する
4
組み込むドメイン知識とベストプラクティスを洗い出す 「エキスパートならどう判断するか」を文言化する

Anthropicが観察した3つのユースケースカテゴリ

カテゴリ1：ドキュメント・アセット作成

プレゼンテーション・アプリ・デザイン・コードなどの成果物を一貫した高品質で作成するために使用します。

実例：frontend-design スキル

"Create distinctive, production-grade frontend interfaces with high design quality. Use when building web components, pages, artifacts, posters, or applications."

主な手法：

  • スタイルガイドとブランド標準の組み込み
  • 一貫した出力のためのテンプレート構造
  • 最終化前の品質チェックリスト
  • 外部ツール不要（Claudeのビルトイン機能を使用）

カテゴリ2：ワークフロー自動化

複数のMCPサーバーにまたがる調整を含む、一貫した手法から恩恵を受けるマルチステッププロセスに使用します。

実例：skill-creator スキル

"Interactive guide for creating new skills. Walks the user through use case definition, frontmatter generation, instruction writing, and validation."

主な手法：

  • バリデーションゲート付きのステップバイステップワークフロー
  • 一般的な構造のテンプレート
  • ビルトインのレビューと改善提案
  • 反復的な改善ループ

カテゴリ3：MCP強化

MCPサーバーが提供するツールアクセスを強化するワークフローガイダンスに使用します。

実例：sentry-code-review スキル（Sentryより）

"Automatically analyzes and fixes detected bugs in GitHub Pull Requests using Sentry's error monitoring data via their MCP server."

主な手法：

  • 複数のMCP呼び出しを順番に調整
  • ドメイン専門知識の組み込み
  • ユーザーが指定しなければならないコンテキストを提供
  • 一般的なMCP問題のエラー処理

アプローチの選択：問題起点 vs. ツール起点

ホームデポに例えると：「キッチンキャビネットを修理したい」という問題を持って行く場合と、新しいドリルを買ってそのドリルの使い方を聞く場合があります。スキルも同様です：

問題起点（Problem-first）

「プロジェクトワークスペースを設定したい」
→ スキルが正しい順序で適切なMCP呼び出しを調整する。ユーザーはアウトカムを説明し、スキルがツールを処理する。

ツール起点（Tool-first）

「Notion MCPに接続している」
→ スキルがClaudeに最適なワークフローとベストプラクティスを教える。ユーザーはアクセス権を持ち、スキルが専門知識を提供する。

ほとんどのスキルはどちらか一方に傾いています。自分のユースケースにどちらのフレーミングが合うかを知ることで、適切なパターンを選べます。

スキルを正しく機能させるための技術的な要件を詳しく解説します。

YAMLフロントマター — スキルの最重要部分

フロントマターはClaudeがスキルを読み込むかどうかを決定する部分です。ここを正しく書くことが最重要です。

最小限の必須フォーマット

---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---

これだけで始められます。

name フィールドのルール

  • kebab-case のみ使用可
  • スペースや大文字は使用不可
  • フォルダ名と一致させること
  • claude または anthropic を名前に含めることは禁止（予約済み）

description フィールドのルール

  • 「スキルが何をするか」と「いつ使うか（トリガー条件）」の両方を必ず含める
  • 1024文字以内
  • XMLタグ（< または >）は使用禁止
  • ユーザーが実際に言いそうな具体的なタスクを含める
  • 関連するファイルタイプがあれば記載する

descriptionの書き方：良い例・悪い例

# 良い例 — 具体的でアクションにつながる
description: Analyzes Figma design files and generates developer
handoff documentation. Use when user uploads .fig files, asks for
"design specs", "component documentation", or
"design-to-code handoff".

# 良い例 — トリガーフレーズを含む
description: Manages Linear project workflows including sprint
planning, task creation, and status tracking. Use when user
mentions "sprint", "Linear tasks", "project planning", or asks
to "create tickets".

# 良い例 — 明確な価値提案
description: End-to-end customer onboarding workflow for PayFlow.
Handles account creation, payment setup, and subscription
management. Use when user says "onboard new customer",
"set up subscription", or "create PayFlow account".

# 悪い例 — 曖昧すぎる
description: Helps with projects.

# 悪い例 — トリガーがない
description: Creates sophisticated multi-page documentation systems.

# 悪い例 — 技術的すぎてユーザートリガーがない
description: Implements the Project entity model with hierarchical
relationships.

セキュリティ制限

フロントマターはClaudeのシステムプロンプトに組み込まれます。悪意あるコンテンツが指示を注入する可能性があるため、以下は絶対に禁止です：
• XMLの角括弧（< >）
• 名前に "claude" または "anthropic" を含めること（予約済み）

スキルはニーズに応じて様々なレベルのテストが可能です。

推奨テストアプローチ

手動テスト（Claude.ai）

クエリを直接実行して動作を観察する。セットアップ不要で高速に反復できる。

スクリプトテスト（Claude Code）

変更をまたいで繰り返し検証できるよう、テストケースを自動化する。

プログラムテスト（Skills API）

定義されたテストセットに対して体系的に実行する評価スイートを構築する。

小さなチームが内部で使うスキルと、数千人のエンタープライズユーザーに展開するスキルでは、テストのニーズが異なります。あなたの品質要件とスキルの公開範囲に合ったアプローチを選んでください。

プロのヒント：1つのタスクを成功するまで反復してから拡張する
効果的なスキル作成者は、1つの難しいタスクをClaudeが成功するまで反復し、その成功アプローチをスキルに抽出します。広範なテストより速いシグナルが得られます。

skill-creator スキルの活用

skill-creator スキルは Claude.ai のプラグインディレクトリから入手でき、Claude Code 用にもダウンロードできます。MCP サーバーがあり、トップ2〜3のワークフローが分かっていれば、1回のセッションで15〜30分以内に機能するスキルを作成・テストできます。

skill-creator でできること

機能

詳細

スキル作成

自然言語の説明からスキルを生成。適切にフォーマットされたSKILL.mdをトリガーフレーズと構造の提案付きで作成

スキルレビュー

一般的な問題（曖昧な説明、トリガーの欠落、構造的な問題）を指摘。スキルの目的に基づいてテストケースを提案

反復改善

エッジケースや失敗例に基づいてスキルを改善する。例：「このチャットで発見された問題を使ってスキルを改善して」

使い方：
「Use the skill-creator skill to help me build a skill for [your use case]」

skill-creator は設計と改善を支援しますが、自動テストスイートの実行や定量的な評価結果の生成は行いません。

フィードバックに基づく改善（Iteration）

スキルはリビングドキュメントです。以下のシグナルに基づいて継続的に改善してください。

アンダートリガーのシグナル

  • スキルが起動すべき場面で起動しない
  • ユーザーが手動で有効化している
  • いつ使うべきか分からないというサポート質問が来る

解決策：descriptionにより多くの詳細とニュアンスを追加する。技術用語などのキーワードを含める。

オーバートリガーのシグナル

  • 無関係なクエリでスキルが読み込まれる
  • ユーザーが無効化している
  • 目的について混乱が生じている

解決策：ネガティブトリガーを追加する。より具体的にする。

実行の問題

  • 一貫性のない結果
  • API呼び出しの失敗
  • ユーザーの修正が必要な場面が多い

解決策：指示を改善し、エラー処理を追加する。

スキルはMCP統合をより完全なものにします。コネクターを比較するユーザーにとって、スキル付きのものはより速く価値を提供できます。

スキルのポジショニング（説明の仕方）

スキルをどう説明するかで、ユーザーが価値を理解して使い始めるかどうかが決まります。

良い例 — アウトカムにフォーカス

「ProjectHubスキルにより、チームは30分の手動セットアップの代わりに、数秒でページ・データベース・テンプレートを含む完全なプロジェクトワークスペースを設定できます。」

悪い例 — 技術的すぎる

「ProjectHubスキルはYAMLフロントマターとMarkdown指示を含むフォルダで、MCPサーバーツールを呼び出します。」

MCPとスキルのストーリーをハイライトする

スキルはClaudeにチームのスプリントプランニングワークフローを教えます。
合わせることで、AI駆動のプロジェクト管理が実現します。」

1. 最終フォーマットを適用
2. サマリーを生成
3. 最終バージョンを保存

主な手法：

スキルの開発中に発生する典型的な問題とその解決策を解説します。

スキルがアップロードできない

エラー：「Could not find SKILL.md in uploaded folder」

原因：ファイル名が正確に SKILL.md ではない

解決策：

  • SKILL.md に正確にリネームする（大文字小文字を区別）
  • ls -la で確認すると SKILL.md と表示されるはず

エラー：「Invalid frontmatter」

原因：YAMLフォーマットの問題

# 間違い — デリミタがない
name: my-skill
description: Does things

# 間違い — クォートが閉じていない
name: my-skill
description: "Does things

# 正しい
---
name: my-skill
description: Does things
---

エラー：「Invalid skill name」

原因：名前にスペースまたは大文字が含まれている

# 間違い
name: My Cool Skill

# 正しい
name: my-cool-skill

スキルがトリガーしない

症状：スキルが自動的に読み込まれない

修正方法：description フィールドを見直してください。

クイックチェックリスト：

  • 説明が曖昧すぎないか？（「プロジェクトを手伝います」は機能しない）
  • ユーザーが実際に言いそうなトリガーフレーズを含んでいるか？
  • 関連するファイルタイプがある場合、記載されているか？

デバッグ方法：Claudeに「When would you use the [skill name] skill?」と聞いてみてください。Claudeがdescriptionを引用します。何が足りないかに基づいて調整します。

スキルがトリガーしすぎる

症状：無関係なクエリでスキルが読み込まれる

解決策1：ネガティブトリガーを追加する

description: Advanced data analysis for CSV files. Use for
statistical modeling, regression, clustering. Do NOT use for
simple data exploration (use data-viz skill instead).

解決策2：より具体的にする

# 広すぎる
description: Processes documents

# より具体的
description: Processes PDF legal documents for contract review

解決策3：スコープを明確にする

description: PayFlow payment processing for e-commerce. Use
specifically for online payment workflows, not for general
financial queries.

MCP接続の問題

症状：スキルは読み込まれるがMCP呼び出しが失敗する

1
MCPサーバーが接続されているか確認する Claude.ai: Settings > Extensions > [Your Service] → 「Connected」と表示されているか
2
認証を確認する APIキーが有効かつ未期限か / 適切な権限・スコープが付与されているか / OAuthトークンが更新されているか
3
MCPを単独でテストする スキルなしでClaudeにMCPを直接呼び出させる。「Use [Service] MCP to fetch my projects」と言ってみる。これが失敗する場合、問題はスキルではなくMCPにある。
4
ツール名を確認する スキルが正しいMCPツール名を参照しているか確認。ツール名は大文字小文字を区別する。MCPサーバーのドキュメントを確認する。

指示が遵守されない

症状：スキルは読み込まれるがClaudeが指示に従わない

よくある原因と解決策

原因

解決策

指示が冗長すぎる

指示を簡潔にする。箇条書きと番号付きリストを使う。詳細な参照は別ファイルに移動する。

重要な指示が埋もれている

重要な指示を先頭に配置する。## Important や ## Critical ヘッダーを使う。必要なら重要ポイントを繰り返す。

言語が曖昧

具体的なアクションを明記する（例：次を確認してからcreate_projectを呼び出す）

# 悪い例 — 曖昧
Make sure to validate things properly

# 良い例 — 具体的
CRITICAL: Before calling create_project, verify:
- Project name is non-empty
- At least one team member assigned
- Start date is not in the past

上級テクニック：重要なバリデーションには、言語指示に頼る代わりに、チェックをプログラム的に実行するスクリプトをバンドルすることを検討してください。コードは決定論的ですが、言語の解釈はそうではありません。Officeスキルでこのパターンの例を参照してください。

モデルの「怠惰」への対処

## Performance Notes
- Take your time to do this thoroughly（徹底的に時間をかけてください）
- Quality is more important than speed（品質はスピードより重要です）
- Do not skip validation steps（バリデーションステップをスキップしないでください）

注：これをSKILL.mdに追加するより、ユーザープロンプトに追加する方が効果的です。

大きなコンテキストの問題

症状：スキルが遅い、または応答が劣化する

原因

  • スキルのコンテンツが大きすぎる
  • 同時に有効なスキルが多すぎる
  • 段階的開示でなくすべてが読み込まれている

解決策

1
SKILL.mdのサイズを最適化する 詳細ドキュメントを references/ に移動する。インライン記述の代わりに参照リンクにする。SKILL.mdを5,000語以内に保つ。
2
有効なスキル数を減らす 20〜50以上のスキルが同時有効かどうか評価する。ロールベースで有効化するスキルを絞り込む。関連する機能をスキル「パック」にまとめることを検討する。

スキルをアップロードする前後に使うチェックリストです。skill-creatorを使って最初のドラフトを生成してから、このリストで漏れがないか確認することを推奨します。

開始前チェック

  • 2〜3つの具体的なユースケースを特定した
  • 必要なツール（ビルトインまたはMCP）を特定した
  • このガイドとサンプルスキルを確認した
  • フォルダ構造を計画した

開発中チェック

  • フォルダ名が kebab-case になっている
  • SKILL.md ファイルが存在する（正確なスペル）
  • YAML フロントマターに --- デリミタがある
  • name フィールドが kebab-case で、スペース・大文字なし
  • description に「何をするか」と「いつ使うか」が含まれている
  • XMLタグ（< >）がどこにも含まれていない
  • 指示が明確でアクションにつながる内容になっている
  • エラー処理が含まれている
  • 例（Examples）が提供されている
  • references/ へのリンクが明確に書かれている

アップロード前チェック

  • 明らかなタスクでのトリガーをテストした
  • 言い換えられたリクエストでのトリガーをテストした
  • 無関係なトピックでは起動しないことを確認した
  • 機能テストが通過した
  • ツール統合が機能している（該当する場合）
  • .zip ファイルとして圧縮した

アップロード後チェック

  • 実際の会話でテストした
  • アンダー・オーバートリガーを監視している
  • ユーザーフィードバックを収集している
  • description と指示を反復改善している
  • メタデータのバージョンを更新した

このチェックリストをコピーして、各スキルの開発チェックシートとして使ってください。

YAML フロントマターの完全なリファレンスです。すべてのフィールドと仕様を網羅しています。

必須フィールド

---
name: skill-name-in-kebab-case
description: What it does and when to use it. Include specific trigger phrases.
---

全オプションフィールド

---
name: skill-name                  # 必須: kebab-case、スペース・大文字禁止

description: |                    # 必須: 1024文字以内、XMLタグ禁止
  スキルが何をするか、いつ使うかを説明。
  具体的なトリガーフレーズを含める。

license: MIT                      # オプション: MIT, Apache-2.0 など

allowed-tools: "Bash(python:*) Bash(npm:*) WebFetch"
  # オプション: ツールアクセスを制限する
  # スペース区切りでツール名またはパターンを指定

metadata:                         # オプション: カスタムキーバリューペア

  version: 1.0.0                  # 推奨: バージョン
  mcp-server: server-name         # 推奨: MCPサーバー名（使用する場合）
  category: productivity          # カテゴリ
  tags:                           # タグ（配列形式）
    - project-management
    - automation
  documentation: https://example.com/docs  # ドキュメントURL
  support: support@example.com    # サポート連絡先

compatibility: |                  # オプション: 1〜500文字
  環境要件を記載（Claude.ai / Claude Code / Claude APIなど）
  必要なシステムパッケージ、ネットワークアクセス要件など

例1：最小構成スキル

まずこれで始めてください。nameとdescriptionの2行だけで動作します。

---
name: meeting-summarizer
description: |
  Generates meeting minutes from transcript text. Creates structured
  summaries with action items, decisions, and next steps.
  Use when user says "summarize this meeting", "create meeting minutes",
  "what did we decide", or pastes meeting transcript text.
---

# Meeting Summarizer

## Instructions

When given meeting transcript or notes, create a structured summary:

### Required Sections

1. **Meeting Overview**
   - Date and participants (if mentioned)
   - Meeting purpose/topic

2. **Key Decisions**
   - List each decision made
   - Format: "DECISION: [what was decided]"

3. **Action Items**
   - Each item with owner and deadline (if mentioned)
   - Format: "ACTION: [task] — Owner: [name] — Due: [date/TBD]"

4. **Next Steps**
   - Upcoming meetings or milestones discussed

### Format Rules
- Use bullet points, not paragraphs
- Keep each point to one sentence
- If information is not in the transcript, omit the section
- Do not invent names, dates, or decisions

## Example Output

**Meeting Overview**
- Topic: Q3 budget review

**Key Decisions**
- DECISION: Increase marketing budget by 15%

**Action Items**
- ACTION: Prepare revised budget sheet — Owner: Finance team — Due: Friday

**Next Steps**
- Follow-up meeting scheduled for next Tuesday

例2：MCP統合スキル（中級）

MCPサーバーと連携する場合の例です。

---
name: linear-sprint-planner
description: |
  Sprint planning workflow for Linear projects. Fetches current issues,
  analyzes team capacity, and creates organized sprint with priorities.
  Use when user says "help me plan the sprint", "organize sprint tasks",
  "create sprint in Linear", "what should we work on this sprint",
  or asks about sprint planning or backlog grooming.

metadata:
  author: YourTeam
  version: 1.0.0
  mcp-server: linear
---

# Linear Sprint Planner

## Prerequisites
- Linear MCP server must be connected
- User must specify: sprint duration (default: 2 weeks) and team

## Instructions

### Step 1: Gather Context
Ask the user for:
- Team name or project identifier
- Sprint start date
- Team capacity (story points or number of engineers)
- Any constraints or priorities

### Step 2: Fetch Current State
Use Linear MCP to:
1. Get all open issues for the project: `linear_get_issues(project_id, status="open")`
2. Get recent sprint history: `linear_get_sprints(project_id, limit=3)`
3. Check team member assignments: `linear_get_team_members(team_id)`

### Step 3: Analyze and Prioritize
Based on the data:
1. Identify high-priority issues (P1, P2)
2. Check estimates vs. capacity
3. Flag any blocking dependencies
4. Consider in-progress work that carries over

### Step 4: Create Sprint Plan
Present a proposed sprint with:
- Selected issues (with reasoning)
- Total estimated capacity used
- Any issues deferred and why

Wait for user approval before creating.

### Step 5: Create Sprint in Linear
After approval:
1. Create sprint cycle: `linear_create_cycle(project_id, name, start_date, end_date)`
2. Add issues to sprint: `linear_add_issues_to_cycle(cycle_id, issue_ids)`
3. Confirm creation with summary

## Error Handling

### Linear MCP Not Connected
If Linear MCP calls fail:
1. Check: Settings > Extensions > Linear shows "Connected"
2. Try refreshing the connection
3. If still failing, ask user to reconnect Linear in settings

### No Issues Found
If no open issues exist:
- Inform user and ask if they want to create new issues or import from another source

### Capacity Overrun
If selected issues exceed capacity:
- Present options: reduce scope, extend sprint, or add capacity
- Never automatically override user's stated capacity

学習のポイント

章

主要なポイント

Chapter 1: Fundamentals

スキル＝フォルダ。必須はSKILL.mdのみ。3段階の段階的開示でトークンを効率化。ポータブルでコンポーザブルな設計。

Chapter 2: Planning

2〜3のユースケースから開始。3カテゴリ（文書作成/ワークフロー自動化/MCP強化）。成功基準を事前に定義する。

Chapter 2: Technical

SKILL.md命名は大文字小文字を区別。nameはkebab-case必須。descriptionにはWHATとWHENの両方を必ず含める。XMLタグ禁止。

Chapter 3: Testing

3種のテスト（トリガー/機能/パフォーマンス比較）。skill-creatorで素早く作成・レビュー。シグナルに基づいて継続改善。

Chapter 4: Distribution

GitHub→Claude.ai/Claude Code でスタート。APIは大規模本番向け。アウトカムベースで説明する。オープンスタンダードなのでポータブル。

Chapter 5: Patterns

5つのパターン（順次/マルチMCP/反復/コンテキスト対応/ドメイン知識）。トラブルシューティングはdescriptionから始める。

Chapter 6: Resources

anthropics/skills リポジトリが公式サンプル。skill-creator が一番の出発点。
