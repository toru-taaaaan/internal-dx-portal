Gartnerの予測では2026年末までにエンタープライズアプリケーションの40%がタスク特化型AIエージェントを組み込みます。市場は2025年の約76億ドルから109億ドルへ拡大し、企業の79%がすでに採用済みです。

しかし多くの企業が直面するのは、「AIエージェントが業務実行能力を持つのに、組織として正しく使えていない」という問題です。

課題1：反復の非効率さ

現状

エージェントは前回の会話を記憶しない。毎回ゼロから詳細な指示を書き直す必要がある。

→

理想

一度定義した指示をどの会話でも即座に呼び出せる。「前回と同じやり方で」が実現する。

課題2：コンテキストウィンドウの圧迫

現状

詳細な指示がトークンを大量消費し、業務データ（ファイル・メール・DB）に使える余地が減少する。

→

理想

必要なときに必要な分だけ読み込む。未使用の指示はコンテキストを一切消費しない。

課題3：品質の属人化

現状

高品質なプロンプトは特定の担当者の頭の中にしか存在せず、チーム間で共有できない。

→

理想

業務知識をファイルとして組織資産化し、Gitで管理・バージョン管理・チーム展開できる。

この3つの課題を解決するために設計されたのが Agent Skills です。次のステップで仕組みを解説します。

Agent Skillsとは、指示・スクリプト・リソースをフォルダにまとめ、AIエージェントが必要なときに必要な分だけ読み込む仕組みです。

2025年10月にAnthropicがClaude向けに公開し、2025年12月にオープンプロトコル（agentskills.io）として仕様を公開。2026年2月時点で26以上のプラットフォームが対応しています。

Skillフォルダの構造

my-skill/               ← Skillフォルダ（任意の名前）
├── SKILL.md            ← 必須：仕様・指示・ワークフロー定義
├── scripts/            ← オプション：実行スクリプト
│   └── process.py
├── references/         ← オプション：参照ドキュメント
│   └── guide.md
└── assets/             ← オプション：画像・テンプレート等
    └── template.xlsx

SKILL.md さえあれば最小構成として動作します。name と description の2フィールドのみが必須です。

SKILL.mdの最小構成

---
name: pdf-extractor
description: |
  PDFファイルからテキストと表を抽出するSkill。
  「このPDFを読んで」「PDFを解析して」と言われたときに使う。
---

# PDF抽出の手順

1. PDFファイルのパスを確認する
2. PyMuPDFを使ってテキストを抽出する
3. 表はpandasのDataFrameに変換して返す

段階的開示（Progressive Disclosure）— Skillsの核心設計

100個のSkillがあっても、コンテキストウィンドウを圧迫しない理由がこれです。

具体例：PDF処理Skillの読み込みフロー

1
ユーザーが「このPDFを読んで」と入力 エージェントはLevel 1の一覧（name/description）をスキャン
2
pdf-extractorがマッチ descriptionの「PDFを読んで」というトリガーキーワードに合致
3
Level 2: SKILL.md本文を読み込む 手順・制約・使用ツールの詳細を取得
4
Level 3: scripts/process.py を読み込む（必要な場合のみ） 実行スクリプトの内容をコンテキストに追加
5
タスクを実行 未使用の99個のSkillはコンテキストを消費しない

100個のSkillでもLevel 1では約3,000〜5,000トークンのみ消費。全文常時読み込みと比べて 50〜100倍のトークン効率化 を実現します。

SKILL.mdはAgent Skillの仕様・指示・ワークフローを定義するMarkdownファイルです。YAMLフロントマター（メタデータ）とMarkdown本文（指示）を1ファイルに統合します。

YAMLフロントマター — 必須フィールド

---
name: skill-name          # 必須：小文字・ハイフン区切り・動名詞形推奨
description: |            # 必須：三人称で「何をするか」＋「いつ使うか」を記述
  このSkillは〇〇を行う。
  「△△して」「××を確認して」と言われたときに使う。
---

悪い例

descriptionの書き方 — なぜ三人称か

descriptionはClaudeがLevel 1スキャン時に読む唯一の情報です。「いつこのSkillを使うか」のトリガーキーワードを必ず含めることが品質を大きく左右します。

区分

記述例

良い例

「このSkillは会議の文字起こしテキストから議事録を生成する。「議事録作って」「ミーティングのまとめ」と言われたときに使う。」

悪い例

「議事録作成。会議ノートを整理します。」（一人称・トリガーなし・曖昧）

Markdown本文 — 4つの記述パターン

パターン1：テンプレート型（厳密な指示）

出力フォーマットを完全に固定したい場合に使う。

# 週次レポートの出力形式

以下のフォーマットで必ず出力すること：

## {週}の業務サマリー
**完了タスク**：{箇条書き}
**進行中**：{箇条書き}
**来週の予定**：{箇条書き}

パターン2：ガイドライン型（柔軟な指示）

Claudeの判断に委ねる部分が多い場合に使う。

# コードレビューのガイドライン

- セキュリティリスクを最優先でチェックする
- パフォーマンスへの影響を評価する
- 可読性の観点からコメントを追加する
- 修正が必要な箇所は具体的な代替案も示す

パターン3：Examples型（入出力例）

期待する出力を例示することで品質を担保する。

# 翻訳スタイルの例

入力：「The deployment was successful.」
出力：「デプロイが正常に完了しました。」

入力：「Please review the attached document.」
出力：「添付ファイルをご確認ください。」

# ルール
- 敬語を使う
- 技術用語はカタカナを優先する

パターン4：Decision Tree型（条件分岐ワークフロー）

状況によって処理を変えたい場合に使う。

# エラー対応フロー

1. エラーメッセージを確認する
   - 「404」→ URLを確認してステップ3へ
   - 「401」→ 認証情報を確認してステップ4へ
   - その他 → ログ全文を表示してステップ5へ

2. ...

複数パターンの組み合わせが標準です。「ガイドライン型 ＋ Examples型」が最も多く使われる組み合わせです。

# ツールの使い方

# 正しい書き方（サーバー名:ツール名 の形式）
mcp__gmail-server__gmail_read を使ってメールを読む

SKILL.mdは500行以内を推奨上限とします。それ以上の情報は references/ ディレクトリに分割して参照パスを記述してください。

評価駆動開発（Evaluation-Driven Development）とは、まず評価シナリオを作り、そこから逆算してSkillを構築・改善する手法です。「作ってからテスト」ではなく「テストを先に定義してから作る」考え方です。

EDD 5ステップ

1
ギャップの特定 「Claudeに任せたいのにうまくいかない」業務を特定する。現状の出力と理想の出力の差を言語化する。
2
評価シナリオの作成（3〜5個） 「このインプットが来たとき、このアウトプットが出れば合格」というテストケースを事前に定義する。
3
ベースライン測定 Skill なしで同じシナリオを実行し、現状のスコアを記録する。
4
最小限のSkillを作成 最初から完璧を目指さない。nameとdescriptionとガイドライン数行だけで始める。
5
反復改善 評価 → ギャップ特定 → Skill修正 → 再評価 のサイクルを繰り返す。

「何を達成したいか（What）は指定し、どう達成するか（How）はClaudeの知能に委ねる」がSKILL.md設計の基本原則です。

Agent SkillsとMCP（Model Context Protocol）は混同されがちですが、役割がまったく異なります。両者を理解して適切に組み合わせることが重要です。

MCPとAgent Skillsの違い

MCP（Model Context Protocol）

エージェントの「手足」

外部サービス・ツールへの接続配線。Gmail・Slack・データベース・APIなど200以上のSaaSをエージェントが操作できるようにする。

業務知識・手順・ノウハウの装備。MCPが「何に接続できるか」を定義するのに対し、Skillsは「どう使うか」を定義する。

Agensにおける3層アーキテクチャ

homulaが提供するAIエージェントプラットフォーム「Agens」では、3層で役割を分担しています。

Layer 1: MCP層

ツール接続（200以上のSaaS接続）。Gmail・Slack・Salesforce・データベースなどへの接続配線。

Layer 2: Agent Skills層

SKILL.md準拠の業務スキルパック。「経費精算」「議事録作成」「週次レポート生成」などの業務ノウハウを装備。

Layer 3: homula独自層

ノーコード管理・配布・権限制御。IT部門がSkillを審査・承認し、部門・ロール別に配布できる管理機能。

実践例：経費精算エージェント

まとめ：MCP = つなぐもの、Skills = やり方、Projects = 知っておくこと。3つを組み合わせることで高品質なエージェントが実現します。

エンタープライズ運用の4次元

1. セキュリティレビュー（必須）

外部から取得したSkillは、展開前に以下の観点で全ファイルを確認します。Snykの調査（2026年2月）では341の悪意あるSkillがマルウェアを配布していたと報告されています。

  • コード実行（scripts/内のファイルが安全か）
  • 指示オーバーライド（既存のCLAUDE.mdルールを上書きしていないか）
  • MCP参照（意図しないツールに接続していないか）
  • ネットワークアクセス（外部通信が発生していないか）
  • ハードコード認証情報（APIキー・パスワードが埋め込まれていないか）

外部から取得したSkillは全ファイルを必ず確認してから展開してください。信頼できるソースからのSkillのみを使用することを推奨します。

2. 評価の義務化

本番展開前に、以下の5つの品質次元でテストします：

評価次元

確認内容

トリガー精度

意図した場面で正しく発動するか

単独動作

他のSkillなしで期待通りに動くか

共存

他のSkillと同時使用時に干渉しないか

指示遵守

SKILL.mdの指示通りに動作しているか

出力品質

期待するフォーマット・内容で出力されるか

3. 運用規模の制約

  • 1リクエストあたり最大 8 Skills まで（Claude API）
  • Skillが多すぎると発動精度が低下するため、ロールベースで配布するSkillを絞り込む

4. Gitを中心としたライフサイクル管理

計画 → 開発 → レビュー → テスト → 展開 → 監視 → 廃止
  ↑_____________________________________|（反復改善）

「Write once, use everywhere」 が設計思想です。1つのSkillフォルダをGitで管理し、Claude.ai・Claude Code・Agent SDK など複数のプラットフォームで再利用できます。

Agent Skillsを組織全体で展開するための12週間のロードマップです。「作って終わり」ではなく、段階的に効果を測定しながら横展開する進め方を推奨します。

12週間フェーズ計画

1
Week 1〜2：業務棚卸し 繰り返し発生する業務タスクを洗い出し、AIエージェントで自動化・標準化できる候補を特定する。担当者ごとに異なるプロンプトを「見える化」する。
2
Week 2〜4：プロトタイプ作成 優先度の高い業務タスクを2〜3個選び、最小構成のSkillを作成。評価シナリオを定義してベースラインと比較する。
3
Week 5〜8：限定展開 プロトタイプの評価スコアが基準を超えたものを、パイロットチームに展開する。実運用でのフィードバックを収集して改善する。
4
Week 9〜12：効果測定・横展開 時間削減・品質向上・コスト削減を定量的に測定し、経営層に報告。効果が実証されたSkillを他部門・他チームに展開する。
