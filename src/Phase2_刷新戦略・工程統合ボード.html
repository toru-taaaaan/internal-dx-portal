追加① ボトルネック分析 — ドライブマッピング（最大の障壁）:
現行GPOで自動割当されている S:ドライブ（\\adsharesvr\public → 全社員利用）と U:ドライブ（\\iotsvr\share → IoT部門利用）はAD（Kerberos認証）に依存。Entra ID完全移行でこの自動接続が動作しなくなる。Azure FilesやSharePoint Onlineへの移行が必要だが、ADSHARESVRには40以上の部門別フォルダに個人名ベースACLが細かく設定されており、クラウド再現が移行プロジェクト最大の工数になる。

追加② GPO移行のIntune代替可否:
ドライブマップ以外のGPO（WinUpdate, スクリーンセーバー, IEモード, RDP, Defender等）はIntuneで代替可能。ドライブマップとLANSCOPEのみ要検討。

追加③ Entra IDに移行しない場合のリスク（4点）:
① MSはADの新機能開発を停止済、MFA・条件付きアクセス等はEntra IDにのみ追加
② Win Serverサポート期限後はパッチ提供終了で脆弱性放置リスク
③ M365/SaaSとの認証統合がAD単体では不可、パスワード二重管理が継続
④ 将来ADサポート完全終了時、準備なしの移行は大規模かつ高リスク

P5
各拠点オンプレサーバーヒアリング
篠部長: 各拠点にどんなサーバーがあるか把握できていない
sec8のネットワーク刷新は本社インフラ（VPN・FW・SD-WAN）中心の議論で、大阪・東北等の拠点にどんなオンプレサーバーが存在するか把握されていなかった。SASE（Cato）導入でSocket配置や通信経路が変わるため、各拠点のサーバー実態（台数・用途・ネットワーク構成）を事前に把握しなければ導入計画が立てられない。拠点側の現状が不明なまま進めるのはリスク。
本社以外の各拠点（大阪・東北等）のオンプレサーバー実態を担当者にヒアリング。①台数 ②用途（ファイル/業務アプリ/プリント等） ③ネットワーク構成を確認し、sec8のCato Socket配置計画・スコープ定義に反映予定 sec8 → 未着手
やること:
本社以外の拠点（大阪・東北等）にどのようなオンプレサーバーが存在するか、現状把握ができていない。

背景: SASE（Cato Networks）導入時に、拠点内サーバーへのアクセスパスが変わる可能性がある。現状のVPN経由アクセスがCato経由に切り替わるため、導入前に各拠点のサーバー実態を把握しておく必要がある。

ヒアリング項目: 各拠点の担当者に対して ①サーバー台数 ②用途（ファイルサーバー/業務アプリ/プリントサーバー等） ③ネットワーク構成（本社との接続方法） を確認する。

反映先: 結果はsec8（SD-WAN/SASE比較）のスコープ定義に反映し、Cato Socket配置計画の精度を上げる。
P6

元のsec8ではIIJ OmnibusとCato SASEを同等の選択肢として並列比較していたが、篠部長が方針を決定。IIJ Omnibusは国内専用バックボーン・各拠点にルーター設置が必要・IIJ AD契約との密結合などの制約があり、Catoのグローバル90+ PoP・Socket1台で完結・ゼロトラスト標準搭載の方が柔軟性が高いと判断。IIJ Omnibus単独シナリオの深掘りに工数を割くのは無駄であり、Cato前提で資料を再構成する指示。
①sec8のSD-WAN比較をCato前提に再構成（IIJ Omnibus単独シナリオを縮小） ②sec9の3パターンをCato統一: LAJ+Cato / USEN+Cato / IIJ AD+Cato ③IIJ一括は参考比較として残すが推奨から外す sec8 → sec9 → 未着手
やること:
篠部長の方針決定 — ネットワーク刷新はIIJ Omnibus（IIJ独自SD-WAN）の深掘りは不要。Cato SASE Cloudを中心に検討を進める。

現状の問題: sec8（SD-WAN比較）にはIIJ Omnibus単独シナリオがまだ含まれている。これをCato＋各ADベンダーの組み合わせパターンに整理し直す必要がある。

具体的な作業:
① sec8のSD-WAN比較表をCato前提に再構成
② sec9（費用全体像）の3パターン比較もCato前提で統一: パターン1（LAJ + Cato）、パターン2（USEN + Cato）、パターン3（IIJ AD + Cato）
③ IIJ一括パターンは「ADもネットワークもIIJ」の参考比較として残すが、推奨からは外す

ベンダー別運用でAH・ADの課題と解決策の妥当性を明記

S2
「ベンダー依存運用が理想」→「オンプレ再構築でいいのでは」のロジック矛盾解消
塩田部長: クラウド移行が必要な理由が弱い。「オンプレ更新でいいのでは」に答えられない
資料はクラウド移行を前提に書かれていたが、「なぜオンプレでサーバーを新しく買い直すのではダメなのか」という当然の反論に対する回答がなかった。新しいハードウェアの方が初期費用は安く、業務フローも変わらない。決裁会議で「オンプレ更新でよくない？」と聞かれた時に論理的に反論できる材料（属人化は解消されない・地理的冗長化がない・24時間監視がない・5〜7年後にまた同じ問題が来る）を資料内に持っておく必要がある。
sec5b末尾に「なぜオンプレ再構築ではなくクラウドか」のロジック補強を追加。新HW購入でも属人化・災害リスク・24時間監視不在は未解決。5〜7年後に再び保守切れ。現状維持5年コスト（¥243万+運用工数）との費用比較を明記 sec5b → 反映済
対応内容: sec5b「追加検討案 まとめ」の末尾に「なぜオンプレミスの再構築ではなくクラウド移行なのか」のロジック補強を追加した。

追加した論旨:
オンプレミスでサーバーを新規購入・再構築する選択肢も考えられるが、それではS1で挙げた「属人化」「災害時の同時喪失リスク」「24時間監視の不在」は解消されない。新しいハードウェアを買っても、保守切れの問題は5〜7年後に再び発生する。

クラウド移行は、ハードウェア更新サイクルからの脱却＋地理的冗長化＋専門家による継続運用を同時に手に入れる手段である。

費用面の反論: 現状維持の5年コスト（サーバー更新¥243万＋運用工数）と比較して、クラウド移行が極端に高額になるわけではない。「オンプレ更新でいいのでは」という反論に対する回答として資料に明記した。
S3
Entra ID: AH・AD移行課題、対応策、移行しない場合のデメリット
塩田部長: Entra IDの説明が一般的すぎる。AKIBAの場合のリスクを具体的に
P2と同様の指摘。Entra IDの機能紹介（MFA・条件付きアクセス等）はあったが、「AKIBAがEntra IDに移行しなかった場合に何が起きるか」のリスクが書かれていなかった。MSがオンプレADの新機能開発を停止している事実、Windows Serverサポート期限後のパッチ終了リスク、M365との認証統合がAD単体では不可能な点など、「移行しない選択肢」のコストを明示しなければ移行の必要性が伝わらない。
P2（篠部長指摘）と統合しsec6に「移行しない場合のリスク」を追加。①MSがAD新機能開発停止 ②Win Serverサポート終了後のパッチ停止 ③M365/SaaSとのSSO不可 ④将来の強制移行時の大規模リスクの4点を明記 sec6 → 反映済
対応内容: P2（篠部長）の対応と統合し、sec6「Entra ID 移行検討」に「Entra IDに移行しない場合のリスク」を追加した。「移行の障壁」（P2のボトルネック分析）と「移行しないリスク」を同じセクションで対比させている。

追加したリスク分析（4点）:
① MicrosoftはオンプレADの新機能開発を停止済み。MFA・条件付きアクセス等の最新セキュリティ機能はEntra IDにのみ追加される
② Windows Serverサポート期限後、パッチ提供が終了し脆弱性が放置されるリスク
③ Microsoft 365・SaaSとの認証統合がAD単体では実現できず、パスワード管理の二重化が続く
④ 将来ADサポートが完全終了した際、準備なしの移行は大規模かつ高リスクになる
S4
費用全体像にパターンごとのメリデメ追加
塩田部長: 金額だけ並べても判断できない。各パターンの利点・欠点を示せ
元のsec9には費用テーブル（初期・月額・5年TCO）が3つ並んでいたが、数字だけでは「安い方を選べばいいのか？」という短絡的な判断しかできない。最安のLAJパターンは自社管理責任が残り、最高額のIIJパターンは全てお任せだがコスト1.5倍。この「コスト vs リスク vs 自由度」のトレードオフを明示しなければ、金額表は意思決定の材料にならない。
sec9の費用テーブル直後に3パターンのメリット・デメリットカードを新設。パターン1: LAJ+Cato（¥40.6M/最安/自社管理残） パターン2: USEN+Cato（¥44.2M/初期最安/SLAなし） パターン3: IIJ一括（¥63.7M/全委託/最高額）のトレードオフを明示 sec9 → 反映済
対応内容: sec9「費用全体像」の費用比較テーブル直後に、3パターンのメリット・デメリットカード（3列レイアウト）を新設した。

パターン1（LAJ + Cato SASE）— 5年TCO ¥40.6M（最安）:
メリット: 5年総額が最も安い / Azure自社名義で将来のEntra ID移行に有利 / サーバー全権限を保持（自社で調査・対応可能）
デメリット: 自社にAD管理責任が残る（属人化リスク継続） / 監視・パッチ適用は別途オプション契約 / SLA未確認

パターン2（USEN + Cato SASE）— 5年TCO ¥44.2M:
メリット: 初期費用が最も安い（¥1.78M） / サーバーRDP接続可能で障害時の直接調査が可能 / 24時間死活監視が標準付属
デメリット: SLAなし（ベストエフォート） / AWS管理コンソールへのアクセス不可 / パッチ適用頻度が低い（四半期〜半年） / USEN名義のためベンダー変更時にデータ移行が必要

パターン3（IIJ 一括）— 5年TCO ¥63.7M（最高額）:
メリット: 全サービスを1社に集約（窓口一本化） / SLA 99.99%の稼働率保証 / パッチ・監視・障害対応すべてIIJ実施 / 属人化リスクを最も軽減
デメリット: 5年総額が最も高い / サーバー直接操作不可（調査・カスタマイズに制限） / IIJ独自基盤のため他社への移行が困難

参考: 現状維持の場合 5年間 ¥12.6M（サーバー更新費¥243万＋ネットワーク月額¥17万×60ヶ月）。ただし地理的冗長化・セキュリティ強化は含まず。
