IIJ（お任せ型）: SLA 99.99%、24時間全自動運用、パッチ適用もIIJ実施。ただし5年TCO最高額（¥11.9M）でサーバー直接操作不可。

P2
Entra ID移行のAKIBA固有ボトルネック
篠部長: 一般論ではなくAKIBAの具体的な障壁を示せ
 GPO台帳・ACL台帳を分析し sec6 に反映 sec6 反映済
反映内容: sec6に「AKIBA固有の移行ボトルネック」セクションを新設。
ボトルネック① ドライブマッピング（最大の障壁）: 現行GPOで自動割当されている S:ドライブ（\\adsharesvr\public → 全社員利用）と U:ドライブ（\\iotsvr\share → IoT部門利用）。 これらはAD（Kerberos認証）に依存しており、Entra ID完全移行で自動接続が動作しなくなる。 Azure FilesやSharePoint Onlineへの移行が必要だが、ADSHARESVRには40以上の部門別フォルダに個人名ベースACLが細かく設定されており、クラウド再現が移行プロジェクト最大の工数。
ボトルネック② GPO移行: ドライブマップ以外のGPO（WinUpdate, スクリーンセーバー, IEモード, RDP, Defender等）はIntuneで代替可能。ドライブマップとLANSCOPEのみ要検討。
Entra ID移行しない場合のリスク: MSはADの新機能開発を停止済。MFA・条件付きアクセス等のセキュリティ機能はEntra IDにのみ追加。ADサーバーのWin Serverサポート期限後はパッチ提供終了で脆弱性放置リスク。M365/SaaSとの認証統合がAD単体では不可でパスワード二重管理が継続。
P3
GPO現状調査
篠部長: GPO管理台帳を共有するので読んでおくこと
 全10シート読込完了 → Intune代替可否表を作成 sec6 反映済
反映内容: 篠部長から共有されたGPO管理台帳（Googleスプレッドシート、10シート）を全シート読込。sec6にIntune代替可否の一覧表を追加。
Intune代替可能 ○: Windows Update制御（→WUfB）、スクリーンセーバー300秒（→デバイス構成プロファイル）、IE Enterprise Mode（→Edge IEモード）、RDP有効化（→Intuneスクリプト）、Defender定義更新（→Endpoint Security）
要検討 △: ドライブマッピング S:/U:（→Intuneスクリプト or Azure Files）、LANSCOPE/位置情報取得（→ベンダー対応状況の確認が必要）
結論: ドライブマップ以外はIntune移行可能。ドライブマップの代替手段が移行計画のクリティカルパス。
P4
ファイルサーバーアクセス権限確認
篠部長: アクセス制御管理台帳を共有するので確認すること
 ADSHARESVR・IOTSVRの権限構造を分析 → sec6に反映 sec6 反映済
反映内容: 篠部長から共有されたアクセス制御管理台帳（Googleスプレッドシート）を読込、sec6のボトルネック分析に反映。
ADSHARESVR（全社共有）: 40+部門フォルダ。HD直下にグループ監査室・コンプライアンス・経営企画室・人事部・総務部等。深いネスト構造（総務専用→取締役会→資料→グループ経営会議）。個人名ベースのフルアクセス権限が多数設定されており、クラウド上での再現が最大工数。
IOTSVR（IoTG専用）: 6フォルダ（sales_極秘、engineer_極秘等）。アクセス権限はIoTグループ＋情シスのみ。比較的シンプル。
結論: ADSHARESVRの個人名ベースACLをAzure Filesで再現する作業が、ファイルサーバー移行の最重要検討事項。
P5
各拠点オンプレサーバーヒアリング
篠部長: 各拠点にどんなサーバーがあるか把握できていない
 各拠点のサーバー台数・用途・構成をヒアリング sec8 未着手
やること: 本社以外の拠点（大阪・東北等）にどのようなオンプレサーバーが存在するか、現状把握ができていない。
SASE（Cato Networks）導入時に拠点内サーバーへのアクセスパスが変わる可能性があるため、導入前に実態を把握しておく必要がある。
各拠点の担当者に対して、サーバー台数・用途（ファイルサーバー/業務アプリ/プリントサーバー等）・ネットワーク構成をヒアリングする。
結果はsec8（SD-WAN/SASE比較）のスコープ定義に反映し、Cato Socket配置計画の精度を上げる。
P6
IIJ深掘り中止、Cato中心に
篠部長: IIJ Omnibusの深掘りは不要。Cato SASEを中心に検討
 sec8/sec9をCato SASE中心に再構成 sec8/9 未着手
やること: 篠部長の方針決定 — ネットワーク刷新はIIJ Omnibus（IIJ独自SD-WAN）の深掘りは不要。Cato SASE Cloudを中心に検討を進める。
現状のsec8（SD-WAN比較）にはIIJ Omnibus単独シナリオが含まれているが、これをCato＋各ADベンダーの組み合わせパターンに整理し直す。
sec9（費用全体像）の3パターン比較もCato前提で再構成：パターン1（LAJ + Cato）、パターン2（USEN + Cato）、パターン3（IIJ AD + Cato）の形に統一する。
IIJ一括パターンは「ADもネットワークもIIJ」の参考比較として残すが、推奨からは外す。

Google Chat（2/16〜17 メッセージ）

発言者: 塩田部長 → 丹治 への具体的修正指示

ID 課題 対応方針 対象 状況
S1
ベンダー別運用でAH・ADの課題と解決策の妥当性を明記
塩田部長: AKIBAの具体的な課題に対して各ベンダーが解決できるか検証せよ
 sec5bに固有課題＋解決度合い比較表を追加 sec5b 反映済
反映内容: sec5bに「AKIBAホールディングス/ADTECのAD運用における現状課題」を新設。
課題① 情シス2名体制の属人化リスク: AD管理権限は実質 丹治・家田の2名。GPO設定・ファイルサーバー権限・DNS設定の全容把握は限定的。退職・長期不在時にAD障害が発生した場合、復旧知識が断絶するリスク。
課題② ハードウェア老朽化と災害リスク: ADサーバー2台が同一フロアに設置。水害・火災時に同時喪失。保守切れ後のハードウェア障害は自社負担（部品調達長期化）。ファイルサーバー（ADSHARESVR/IOTSVR）も同一サーバー室で、AD障害時にファイルアクセスも連鎖停止。
各社の課題解決度合い（5項目比較表）: 属人化解消 IIJ◎全委託/USEN△OS管理は自社/LAJ△管理権限は自社。地理的冗長化 全社◎。保守切れ解消 全社◎。24時間監視 IIJ◎標準/USEN◎標準/LAJ○オプション。SLA IIJ◎99.99%/USEN×なし/LAJ△確認中。
S2
「ベンダー依存運用が理想」→「オンプレ再構築でいいのでは」のロジック矛盾解消
塩田部長: クラウド移行が必要な理由が弱い。オンプレ更新でいい反論に答えられない
 sec5bまとめにロジック補強を追加 sec5b 反映済
反映内容: sec5bのまとめに「なぜオンプレ再構築ではなくクラウド移行なのか」のロジックを追加。
論旨: オンプレでサーバーを新規購入・再構築しても、S1で挙げた「属人化」「災害時の同時喪失リスク」「24時間監視の不在」は解消されない。 新しいハードウェアを買っても、保守切れの問題は5〜7年後に再び発生する。 クラウド移行は、ハードウェア更新サイクルからの脱却＋地理的冗長化＋専門家による継続運用を同時に実現する手段。
費用面の反論: 現状維持の5年コスト（サーバー更新¥243万＋運用工数）と比較して、クラウド移行が極端に高額になるわけではない。
S3
Entra ID: AH・AD移行課題、対応策、移行しない場合のデメリット
塩田部長: Entra IDの説明が一般的すぎる。AKIBAの場合のリスクを具体的に
 P2と統合。sec6にリスク分析を追加 sec6 反映済
反映内容: P2の対応と統合し、sec6に「Entra IDに移行しない場合のリスク」を追加。
リスク①: MicrosoftはオンプレADの新機能開発を停止済み。MFA・条件付きアクセス等のセキュリティ機能はEntra IDにのみ追加される。
リスク②: Windows Serverサポート期限後、パッチ提供が終了し脆弱性が放置されるリスク。
リスク③: M365・SaaSとの認証統合がAD単体では実現できず、パスワード管理の二重化が続く。
リスク④: 将来ADサポートが完全終了した際、準備なしの移行は大規模かつ高リスクになる。
これらをsec6のボトルネック分析（P2）と一体で記載し、「移行の障壁」と「移行しないリスク」を対比させた。
S4
費用全体像にパターンごとのメリデメ追加
塩田部長: 金額だけ並べても判断できない。各パターンの利点・欠点を示せ
 sec9にメリデメカードを追加 sec9 反映済
反映内容: sec9の費用テーブル直後に3パターンのメリット・デメリットカードを新設。
パターン1（LAJ + Cato）5年TCO ¥40.6M 最安: Azure自社名義でEntra ID移行に有利、全権限保持。ただし自社にAD管理責任が残り属人化リスク継続。監視・パッチ適用は別途オプション。
パターン2（USEN + Cato）5年TCO ¥44.2M: 初期費用最安¥1.78M、RDP接続可で障害時に直接調査可能、24時間死活監視標準。ただしSLAなし、AWS管理コンソール不可、パッチ適用は四半期〜半年。USEN名義のためベンダー変更時にデータ移行が必要。
パターン3（IIJ一括）5年TCO ¥63.7M 最高額: 全サービス1社集約で窓口一本化、SLA 99.99%、パッチ・監視・障害対応すべてIIJ実施で属人化リスク最軽減。ただしサーバー直接操作不可、IIJ独自基盤で他社移行が困難。
参考: 現状維持5年間 ¥12.6M（ただし地理的冗長化・セキュリティ強化は含まず）
