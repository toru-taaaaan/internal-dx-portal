ADサーバーとは

Active Directory（AD）は、社員がPCにログインする際に「この人は本当に社員か」「どのファイルにアクセスしてよいか」を確認する仕組みです。 社員がPCを起動してログインするたびに、このサーバーに問い合わせが行われます。

現在、ADサーバーはプライマリ（PDCSVR）とセカンダリ（BDCSVR）の2台構成です。 片方が故障しても、もう片方が稼働していればPCログインは可能です。 ただし、2台とも築地本社6階の同じサーバー室に設置されているため、火災・地震等で同時に被害を受ける可能性があります。

SECOMサーバーとは

各拠点のドアに設置されたカードリーダーと連携し、誰がいつどこに入退室したかを管理するサーバーです。 各拠点にインターフェース機器が設置されており、入退室の認証自体は拠点側で動作します。 こちらは1台構成で、予備機はありません。

また、このサーバーは勤怠管理システムとも連携しており、最初の入室を「出勤」、最後の退室を「退勤」として自動打刻する仕組みです。 サーバーが停止した場合でも、各拠点の入退室自体は拠点側の機器で継続できます。 ただし、勤怠管理システムとの連携が停止します。

現在の構成を踏まえ、このまま使い続けた場合にどのような状況になるかを整理します。

このまま使い続けた場合

現在の構成は、導入当時の一般的な設計です。 ADサーバーは2台構成（PDC + BDC）のため、片方の故障時にはもう片方で業務継続が可能です。 ただし、2台とも同じサーバー室（築地6階）に設置されており、地理的な冗長性はありません。 SECOMサーバーは1台のみで、予備機がない状態です。 なお、ADサーバーは2020年に導入され、標準の5年保守は既に終了しています。現在は延長保守中で、2027年10月末が期限です。 SECOMサーバーも延長保守中ですが、導入時期・保守期限はADサーバーとは異なります。

設置場所について

これらのサーバーは、本社6階の事務所の一角に設けられたサーバー室に設置されています。 専用データセンターでは、耐震・耐火設備、電源の冗長化（自家発電・UPS）、専用空調（24時間365日の温度・湿度管理）、防水対策（漏水検知・二重床）、高性能フィルターによる埃の除去、厳格な入退室管理などが整備されています。 事務所内のサーバー室は、これらの設備を持たず、オフィスビルの一般的な設備に依存しています。空調は業務用エアコンで、夜間や休日に停止する場合があり、上階からの漏水対策もありません。 クラウドへ移行した場合、サーバーはベンダーのデータセンター（国内複数拠点に分散）で運用されるため、設置場所に起因するリスクは解消されます。

水害リスク（漏水）

サーバー室が事務所ビルの一角にある場合、上階からの漏水（給排水管の破損、空調ドレンの詰まり等）によりサーバーが損壊するリスクがあります。 専用データセンターでは防水対策（二重床、漏水検知センサー、排水設備）が整備されていますが、一般のオフィスビルではこうした対策は行われていません。

温度・空気環境の違い

サーバーは高温・高湿度・埃に弱い精密機器です。 専用データセンターでは、年間を通じて室温22〜25℃・湿度40〜50%に維持する専用空調が常時稼働し、高性能フィルターで埃を除去しています。 一方、事務所内のサーバー室は、業務用エアコンに依存しており、夜間・休日に空調が停止する場合があります。 また、事務所と同じ空気を共有するため、埃や紙粉がサーバー内部に蓄積し、故障の原因となることがあります。 クラウドへ移行した場合、こうした物理環境に関するリスクは全てデータセンター側の管理となります。

以上を踏まえ、各サーバーをクラウドへ移行する案を検討しました。まず、影響範囲の大きいADサーバーから説明します。

ADサーバーのクラウド移行

現在のADサーバーは2台構成で、片方の故障には対応できます。 ただし、同一フロアにあるため災害時のリスクは残ります。 クラウドに移すことで、地理的に分散されたデータセンターの冗長構成を活用でき、災害時のリスクを軽減できます。 また、ハードウェアの保守切れを気にする必要がなくなります。

3社から提案を受けました。それぞれの特徴は、「サーバーの預け方」が大きく異なります。

IIJ
お任せ型
サーバーの管理を全てIIJに任せる方式。自社ではADの操作画面だけを使う。サーバー本体には触れない。パッチ適用や監視は全てIIJが実施。
USEN
分離型
AWS上に仮想サーバーを設置し、OSとADの管理は自社で行う方式。AWSのアカウントはUSEN名義。サーバーにリモート接続して操作可能。
Link-At-Japan
並走型
Azure上に自社名義のアカウントで環境を構築し、Link-At-Japanが技術支援する方式。管理コンソールは共同で操作。最も自由度が高い。

ベンダー別：日常運用でできること

各社を選んだ場合に、情報システム部がクラウド上のADサーバーに対して「何ができて、何ができないか」を整理します。 日常のユーザー追加やパスワードリセットだけでなく、障害時の調査やトラブル対応など、実務で必要になる操作の可否をまとめました。

IIJを選んだ場合（お任せ型）

自社でできること
  • ユーザーの追加・削除・パスワードリセット（専用管理画面またはRSAT経由）
  • グループポリシー（GPO）の作成・変更
  • OUの作成・編集
  • DNS/DHCPの基本操作
自社ではできないこと
  • サーバーへのリモートデスクトップ（RDP）接続
  • OSの設定変更（レジストリ、イベントログ詳細設定等）
  • 任意のソフトウェアのインストール
  • ADスキーマの拡張
  • AWS/Azure管理コンソールへのアクセス（IIJ独自基盤のため該当なし）

障害時の対応イメージ: ADに障害が発生した場合、IIJが検知・対応・復旧まで全て実施する。情シスへは事後報告。自社で調査のためにサーバーにログインすることはできないが、その必要もない。IIJ側で24時間365日監視しており、自動切替（Active/Active構成）で瞬断〜分単位での復旧が見込める。

USENを選んだ場合（分離型）

自社でできること
  • サーバーへのRDP接続（VPN経由、標準2セッション）
  • ユーザー・GPO・OU・DNS/DHCPの全操作
  • OS内部の設定変更（イベントログの確認、レジストリ編集等）
  • 障害発生時にサーバーにログインして直接調査
自社ではできないこと
  • AWS管理コンソールへのアクセス（Route53、VPC、IAM、CloudWatch等すべて不可）
  • サーバーの停止・再起動（USENへ依頼が必要）
  • ネットワーク構成の変更（VPC、セキュリティグループ等）
  • OS管理者権限の有無 → 確認中（2/14問合せ済）

障害時の対応イメージ: USENが死活監視・リソース監視を24時間実施しており、異常を検知した場合は連絡が入る。既知の障害はUSEN側で自動復旧。未知の障害は調整の上で対応。情シスもRDPでサーバーにログインし、イベントログ等を直接確認して原因調査が可能。ただし、AWS基盤側の問題（ネットワーク、ストレージ等）はUSENへの依頼が必要で、SLAはない（ベストエフォート対応）。

Link-At-Japanを選んだ場合（並走型）

自社でできること
  • サーバーへのRDP接続（VPN経由）
  • ユーザー・GPO・OU・DNS/DHCPの全操作
  • Azure Portal（管理コンソール）への直接アクセス
  • OS内部の設定変更・直接調査
  • サーバーの停止・再起動
  • 解約時のデータ取り出し（自社名義のため自由）
ベンダーが代行してくれること
  • 24時間365日の監視（Zabbix / Azure Monitor）
  • 障害発生時の一次対応・復旧（アシストプラン: ¥50,000/月）
  • Windows Updateの適用代行（アシストプラスプラン: ¥68,000/月）
  • Azure英語通知の日本語変換・連絡
  • Azure基盤側の構成変更支援

障害時の対応イメージ: Link-At-Japanが24時間365日監視しており、異常検知時は一次対応と復旧を実施。情シスもRDPやAzure Portalから直接状況を確認できる。自社名義のAzure環境のため、ログの確認、リソースの状態確認、ネットワーク設定の確認など、全ての情報に自社でアクセス可能。ベンダーと同じ画面を見ながら対応を進められる。

自社で管理権限を持つことのリスク

サーバーに直接アクセスできることは自由度が高い反面、それに伴うリスクも発生します。 管理コンソールやサーバーへのアクセス権限は、裏を返せば「システムを止められる権限」でもあります。

管理権限の漏洩・不正利用
  • 管理者アカウントが漏洩した場合、全社員のPCログインに影響する障害を引き起こされる可能性がある
  • 大手企業の情シス部門の51.3%が情報漏洩の危機を経験しており、「アクセス権限の誤設定」が原因の17.7%を占める（Assured社 2023年調査）
  • 退職者のアカウントが削除されず残っている場合、元社員による不正アクセスのリスクがある
操作ミスと属人化
  • 管理者権限での操作ミス（設定変更、パッチ適用の失敗等）は、全社員のPC認証に直接影響する
  • 担当者が退職した場合、サーバーの設定内容やパスワードがブラックボックス化し、引き継ぎが困難になる
  • 中堅企業の約38%が情シス担当者1人以下で運用しており、その担当者の離職がシステム全体の機能不全を招く（Dell EMC IT投資動向調査）
学習コストとセキュリティ対応の限界
  • クラウド環境（AWS / Azure）の管理には専門知識が必要であり、担当者の学習・習熟に相応の時間がかかる
  • サイバー攻撃の手法は日々高度化しており、ランサムウェアやゼロデイ攻撃への対応を自社の少人数で行うことには限界がある
  • クラウドの特権IDは、オンプレミス以上に攻撃者に狙われやすく、アクセス経路の多さから管理の難易度が高い（NRIセキュアテクノロジーズ）

専門事業者に運用を委託する意義

上記のリスクに対し、マネージドサービス（運用代行）を活用することで、以下の効果が期待できます。

契約によるリスク移転
  • SLA（稼働率保証）により、障害時の対応責任と復旧目標がベンダーとの契約で担保される
  • 操作ミスや設定不備の責任がベンダー側に移転するため、自社の担当者個人にリスクが集中しない
  • 担当者が退職してもベンダーの運用体制は変わらず、業務継続性が確保される
専門家による継続的な防御
  • セキュリティパッチの適用判断や脆弱性対応を、最新の知見を持つ専門家が継続的に実施する
  • 24時間365日の監視により、異常の早期検知と迅速な対応が可能になる
  • 自社の担当者がクラウド管理の学習に時間を割く必要がなく、本来の業務（社内のIT活用推進等）に集中できる

まとめ: サーバーに自由にアクセスできることは、技術的には柔軟性が高い構成です。 一方で、管理権限を自社で持つことは、その権限の漏洩・誤操作・属人化といったリスクも同時に引き受けることになります。 ADサーバーは全社員のPC認証を担う基盤であり、障害が発生すると全拠点の業務が停止します。 このような重要基盤の運用管理を、契約に基づくSLAと専門家の体制で守ることは、経営上のリスク管理として合理的な選択です。

管理体制を踏まえ、次にMicrosoftのクラウドID管理サービス「Entra ID」について調査した結果を報告します。

Entra IDについて

ADの移行を検討するにあたり、MicrosoftのクラウドID管理サービス「Entra ID（旧Azure AD）」について調査しました。 AKIBAホールディングスでも、Office 365の認証にはすでにEntra IDが使われています。 今後の基盤を考えるうえで、ADとEntra IDの関係を整理します。

Office 365とEntra IDの関係: Office 365（Microsoft 365）を契約した時点で、Entra IDのテナント（認証基盤）は自動的に作成されています。 社員がOutlookやTeamsにログインするたびに、裏側ではEntra IDが認証を処理しています。 Office 365の管理者アカウントで https://entra.microsoft.com にアクセスすると、登録されている全ユーザーを確認できます。

ADとEntra IDの役割の違い

ADとEntra IDは、名前は似ていますが役割が異なります。

Active Directory
社内ネットワークの管理
  • 社員のPCログイン認証
  • ファイルサーバーへのアクセス権管理
  • PCの設定を一括で制御（グループポリシー）
  • インターネットがなくてもログイン可能
≠
Microsoft Entra ID
クラウドサービスのID管理
  • Office 365等へのログイン管理
  • 多要素認証（なりすまし防止）
  • 条件付きアクセス（場所・端末による制限）
  • インターネット接続が必要

Entra IDだけでは対応できないこと

現時点では、Entra IDだけでADの全機能を代替することはできません。以下の機能はADに依存しています。

  • ファイルサーバーへの自動ログイン（Kerberos認証）
  • PCの詳細な設定制御（グループポリシー）
  • インターネット未接続時のPCログイン

これらを補うには、Microsoft Intuneという別のサービスの契約が必要であり、さらに社内の全アプリケーションがクラウドに対応している必要があります。

ADにはできず、Entra IDで初めて使える機能

ADとEntra IDは同じMicrosoftの製品ですが、Entra IDにはAD単体では実現できないセキュリティ機能があります。 ハイブリッド構成にする最大の理由は、これらの機能を使えるようにするためです。

多要素認証（MFA）

パスワードに加え、スマートフォンの認証アプリや生体認証を要求する。 パスワードが漏洩しても、第三者はログインできない。 ADにはこの機能がない。

条件付きアクセス

「会社のPCからのみ許可」「海外からのログインは拒否」「私物スマホからはメール閲覧のみ」など、 状況に応じたアクセス制御ができる。 ADにはこの機能がない。

不正ログインのAI検知

普段と異なる場所・時間・端末からのログインをAIが自動検知し、追加認証を要求する。 不正アクセスの兆候を人手に頼らず検出できる。 ADにはこの機能がない。

パスワードレス認証

Windows Hello（顔認証・指紋認証）やセキュリティキーでパスワード自体を不要にする。 パスワードが存在しないため、漏洩リスクそのものがなくなる。 ADにはこの機能がない。

つまり、ハイブリッド構成にする目的は「ADを捨てる」ことではなく、「ADの管理機能を維持したまま、Entra IDのセキュリティ機能を追加で使えるようにする」ことです。 なお、多要素認証や条件付きアクセスはMicrosoft 365 Business Premiumに含まれており、追加ライセンス費用は発生しません（現在のライセンスプランによる）。

将来像: Entra IDへの集約は目指すべきか

Microsoftは、ID管理に関する新機能開発を全てEntra IDに集中させており、ADへの新機能投資は事実上停止しています。 Gartner社のAccess Management分野の評価では、Microsoftは9年連続でリーダーに選出されています（2025年11月）。 長期的にはEntra IDに認証基盤を集約する方向が業界の標準的な流れです。

ただし、Microsoftは「ADを廃止する予定はない」と明言しています。 Windows Server 2025でもADの機能は維持されており、サポートは2034年まで継続します。 現時点では大手企業の約80%がADとEntra IDを併用する「ハイブリッド構成」を採用しており、 Microsoftも段階的な移行（ハイブリッド → クラウド集約）を推奨しています。 オンプレミスADからEntra IDへの直接的な移行パスは存在せず、一気にADを廃止する方法は推奨されていません。

Entra IDのみに移行した場合の追加の制限事項: グループポリシー（GPO）の代替となるIntuneは、GPOと同等の粒度での設定制御には対応しきれていない部分がある。 プリンターはUniversal Print対応機器またはコネクタが必要。 また、Entra ID参加デバイスは初回ログイン時にインターネット接続が必須となる。

ネットワークの課題と刷新案

現在、各拠点のインターネット通信は、一度築地本社を経由してから外部に接続する構成になっています。 特にGoogle MeetによるWeb会議で、映像の遅延や音声の途切れが発生しています。Office 365はアカウント管理として利用しており、業務の中心はGoogle Workspaceです。Web会議の利用が増えるにつれ、全拠点のトラフィックが築地本社を経由する構成が問題になっています。

また、現在使用しているVarioSecure社のVPN機器は、セキュリティ面での制約があり、テレワーク環境への対応も限定的です。

VPN・SD-WAN・SASEの違い

現行
VPN（VarioSecure）
各拠点にルーターを設置し、暗号化されたトンネル（VPN）で本社と接続する仕組み。全てのインターネット通信が本社を経由するため、本社の回線に負荷が集中する。セキュリティ機能はルーターに依存。
SD-WAN
IIJ Omnibus
ソフトウェアで通信経路を自動制御する仕組み。各拠点から直接インターネットに接続できるため、本社への負荷集中を解消できる。Google MeetやMicrosoft 365などは拠点から直接接続（ローカルブレイクアウト）。セキュリティはクラウド型UTM（ファイアウォール）で確保。
SASE
Cato
SD-WANの機能に加え、ファイアウォール、不正侵入検知、ウイルス対策、URLフィルタ、SSL検査などのセキュリティ機能をクラウド上で一体提供する仕組み。拠点側にはSocket（小型機器）1台を設置するだけ。リモートアクセスも標準機能として含まれる。
