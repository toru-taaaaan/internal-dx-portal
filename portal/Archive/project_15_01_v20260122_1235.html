📋 ベンダー側の実作業（責任範囲）の想定

「手作業」か「LANScope」かに関わらず、ベンダーが責任を持つべき作業工程を定義します。

IIJ (フルマネージド案)
【ベンダー作業：重】
  • AD基盤(PaaS)の構築・起動
  • オンプレADとの同期ブリッジ設定
  • Managed DNSによる一括切替制御
  • 移行後のADディレクトリ正常性監視
  • ※社内作業はほぼなし
USEN (IaaS / 並行稼働案)
【ベンダー作業：中】
  • AWS EC2(Windows)基盤の構築
  • 拠点間VPN(VSR連携)の設定・試験
  • オンプレADとのレプリケーション構成
  • 新DNSアドレス等の「設定値」情報の提供
  • ※切替コマンド( logic )の提示
LAJ (IaaS / 計画停止案)
【ベンダー作業：中強】
  • Azure VM(東西2拠点)基盤の構築
  • 拠点間VPNおよびルーティング設定
  • 休日におけるデータ移行・切替実作業
  • 切替後のサーバー側疎通確認
  • クライアント配布情報の作成・提供

確実性を担保するための「内製検証」ステップ (案)

Q1. LANScopeで実際に何ができるか？

ベンダー選定「前」に、社内で1〜2台のテストPCを使い、LANScopeからDNS設定変更コマンドが確実に配布・反映されるかを検証します。これにより「ベンダー任せにしない技術的基盤」を自社で確認できます。

Q2. ベンダーに何を投げれば負担が減るか？

全ベンダーに詳細計画を求めるのではなく、比較表を元に「この手順で合っているか？」という「Yes/No」での確認に留めます。詳細手順は、内定した1社に対して「設定値のロジック」のみを要請することで、双方の工数を最適化します。
